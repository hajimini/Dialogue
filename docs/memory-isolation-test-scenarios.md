# 记忆隔离测试场景设计

## 测试目标

验证系统在不同角色（Character）和人设（Persona）组合下的记忆隔离机制，确保记忆不会跨边界泄露。

---

## 核心隔离维度

系统应该在以下维度实现记忆隔离：

1. **Character 隔离**：同一用户与同一人设对话时，不同角色的记忆应该完全隔离
2. **Persona 隔离**：同一用户与不同人设对话时，记忆应该按人设隔离
3. **组合隔离**：Character + Persona 的组合应该形成独立的记忆空间

---

## 测试场景

### 场景 1：同用户 + 同人设 + 不同角色

**目标**：验证角色级别的记忆隔离

**设置**：
- 用户：Admin
- 人设：小芮嫣
- 角色 A：直连A-082698
- 角色 B：直连B-082698

**测试步骤**：

1. **角色 A 对话**：
   - 发送消息："我最喜欢的颜色是红色"
   - 发送消息："我的宠物叫小白"
   - 预期：系统存储 `user_fact: 喜欢红色` 和 `user_fact: 宠物叫小白`，关联到 `character_id=角色A`

2. **角色 B 对话**：
   - 发送消息："我最喜欢的颜色是蓝色"
   - 发送消息："我的宠物叫小黑"
   - 预期：系统存储 `user_fact: 喜欢蓝色` 和 `user_fact: 宠物叫小黑`，关联到 `character_id=角色B`

3. **验证隔离**：
   - 查询角色 A 的记忆：应该只返回"红色"和"小白"
   - 查询角色 B 的记忆：应该只返回"蓝色"和"小黑"
   - 角色 A 的对话上下文中不应该出现"蓝色"或"小黑"
   - 角色 B 的对话上下文中不应该出现"红色"或"小白"

**预期结果**：✅ 记忆完全隔离，无交叉污染

### 场景 4：无角色 vs 有角色

**目标**：验证无角色（character_id=null）和有角色的记忆隔离

**设置**：
- 用户：Admin

2. **有角色对话**：

3. **验证隔离**：

   - 有角色对话上下文中不应该出现"3 月 15 日"

**预期结果**：✅ null 和非 null character_id 的记忆完全隔离

## 验证方法

### 1. API 查询验证

使用 `/api/memories` 端点查询记忆：

```bash
# 查询特定角色的记忆
GET /api/memories?persona_id={persona_id}&character_id={character_id}

# 查询特定人设的所有记忆
GET /api/memories?persona_id={persona_id}

# 查询特定角色的所有记忆（跨人设）
GET /api/memories?character_id={character_id}
```

### 2. 数据库直接查询

```sql
-- 查询特定组合的记忆
SELECT * FROM memories
WHERE user_id = 'admin_user_id'
  AND persona_id = 'persona_a_id'
  AND character_id = 'character_a_id';

-- 验证记忆隔离
SELECT character_id, persona_id, COUNT(*) as memory_count
FROM memories
WHERE user_id = 'admin_user_id'
GROUP BY character_id, persona_id;
```

### 3. 对话上下文验证

发送测试消息后，检查 AI 回复中是否包含不应该出现的记忆内容：

```typescript
// 发送消息到角色 A
const responseA = await sendMessage({
  personaId: 'persona_a_id',
  characterId: 'character_a_id',
  message: '你还记得我最喜欢的颜色吗？'
});

// 验证回复中只包含角色 A 的记忆（红色），不包含角色 B 的记忆（蓝色）
assert(responseA.includes('红色'));
assert(!responseA.includes('蓝色'));
```

## 失败场景

如果测试失败，可能的原因：

1. **记忆查询未过滤 character_id**：
   - 症状：查询角色 A 的记忆时，返回了角色 B 的记忆
   - 原因：`match_memories` 函数或 API 查询未正确过滤 `character_id`

2. **记忆存储未记录 character_id**：
   - 症状：所有记忆的 `character_id` 都是 null
   - 原因：`mem0-adapter.ts` 中的 `add()` 方法未传递 `characterId` 参数

3. **对话上下文未隔离**：
   - 症状：AI 回复中包含其他角色的记忆
   - 原因：`getContext()` 方法未正确过滤 `character_id`

4. **会话级别的 character_id 未传递**：
   - 症状：同一会话中切换角色后，记忆仍然关联到旧角色
   - 原因：会话表的 `character_id` 未更新，或 API 未从会话中读取 `character_id`

---

## 成功标准

所有场景的测试都应该通过以下验证：

✅ 记忆存储时正确记录 `character_id` 和 `persona_id`
✅ 记忆查询时正确过滤 `character_id` 和 `persona_id`
✅ 对话上下文中只包含当前组合的记忆
✅ 不同组合之间的记忆完全隔离，无交叉污染
✅ null 和非 null `character_id` 的记忆正确隔离

---

## 自动化测试建议

1. **使用 Jest 或 Vitest** 编写自动化测试
2. **测试前清理数据**：删除测试用户的所有记忆
3. **使用固定的测试数据**：确保测试可重复
4. **并行测试隔离**：每个测试使用独立的用户/角色/人设组合
5. **详细的断言**：不仅验证记忆存在，还要验证记忆内容和关联的 ID
6. **日志记录**：记录每个测试步骤的 API 请求和响应，便于调试

---

## 测试执行顺序

1. 运行场景 1（角色隔离）
2. 运行场景 4（null vs 非 null）
3. 运行场景 2（人设隔离）
4. 运行场景 3（组合隔离）

建议按此顺序执行，因为场景 1 和 4 是最基础的隔离测试，场景 2 和 3 是更复杂的组合测试。
